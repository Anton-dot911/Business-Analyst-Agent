# ТЗ №2 — Business Analyst Agent

**Кодова назва:** Insight
**Тип:** портфоліо-продукт
**Пріоритет:** другий (після DocFlow)
**Оцінка тривалості:** 3 тижні

---

## 1. Проблема та бізнес-кейс

Керівник малого бізнесу має дані у БД (продажі, клієнти, склад) і документи (політики, договори, інструкції), але для відповіді на просте питання ("який маржинальний продукт цього кварталу?", "які умови повернення у нас прописані?") потрібен аналітик або ручний пошук.

**Рішення:** агент, що приймає питання природною мовою і сам вирішує, звідки брати відповідь: **SQL** (кількісні питання до БД), **RAG** (питання до внутрішніх документів), **Web search** (зовнішній контекст — курси, ринок), або **комбінація**. Відповідь — текст з графіком (де доречно) і посиланнями на джерела.

Це продуктова реалізація архітектури semantic router над кількома бекендами (агентний RAG з інтелектуальним роутингом запитів), яку раніше опрацьовано концептуально.

**Що демонструє:** agentic-патерни (routing, tool use, self-correction), text-to-SQL з безпекою, RAG з цитуванням, синтез з кількох джерел.

## 2. Користувачі та сценарії

- **Керівник/менеджер:** ставить питання, отримує відповідь з графіком і джерелами, ставить уточнення (діалог зі збереженням контексту).
- **Демо-режим:** передзаповнена демо-компанія (роздрібна торгівля: ~10 тис. рядків продажів за 2 роки, 15 документів політик) + 8 підказок-прикладів питань.

**Ключові сценарії:**
1. "Порівняй виторг по категоріях за Q1 і Q2" → router: SQL → графік bar chart + текстовий висновок.
2. "Яка у нас політика повернення для юросіб?" → router: RAG → відповідь з цитатами і посиланням на документ.
3. "Наші продажі електроніки vs загальний тренд ринку?" → router: SQL + Web → синтезована відповідь з розділенням внутрішніх/зовнішніх даних.

## 3. Функціональні вимоги

### MVP
- **F1. Чат-інтерфейс:** діалог з історією, streaming відповіді, підказки-приклади.
- **F2. Semantic Router:** класифікація запиту → один або кілька маршрутів (`sql | rag | web | clarify`). Маршрут `clarify` — коли запит неоднозначний: агент ставить уточнювальне питання замість вгадування.
- **F3. SQL-маршрут:** генерація SQL за схемою БД → виконання під read-only роллю → якщо помилка виконання, одна ітерація self-correction (помилка повертається моделі) → результат.
- **F4. Візуалізація:** якщо результат SQL табличний і придатний для графіка — агент обирає тип (line/bar) і повертає специфікацію; фронтенд рендерить (Recharts).
- **F5. RAG-маршрут:** пошук по pgvector (документи компанії, чанкінг по секціях) → відповідь строго на основі знайденого, з інлайн-цитатами `[1]`, `[2]` і списком джерел. Якщо релевантного контенту немає — чесне "у документах відповіді немає".
- **F6. Web-маршрут:** web search tool через Claude API; зовнішні дані у відповіді явно марковані як зовнішні, з URL.
- **F7. Прозорість (диференціатор для портфоліо):** розгортуваний блок "How I answered": обраний маршрут, згенерований SQL / знайдені чанки / пошукові запити, час і вартість кроків.
- **F8. Демо-дані:** сідер демо-компанії (SQL + документи), одна команда розгортання.

### v1.1
- F9. Збережені дашборди: закріплення відповіді-графіка, авто-оновлення при відкритті.
- F10. Пам'ять фактів компанії між сесіями (наприклад, "наш фінансовий рік починається у березні").
- F11. Експорт відповіді у PDF-звіт.

### Out of scope
Write-операції в БД; підключення довільних БД користувача через UI (тільки конфіг); прогнозування/ML.

## 4. Нефункціональні вимоги

- Перший токен відповіді ≤ 3 с; повна відповідь простого SQL-питання ≤ 10 с (p95).
- Routing accuracy ≥ 92% на golden dataset.
- SQL: виконуваність згенерованих запитів ≥ 90% з першої спроби, ≥ 97% після self-correction.
- RAG: groundedness — 0 тверджень без підтвердження у знайдених чанках (ручна перевірка на eval-наборі).
- Безпека: read-only роль, allowlist таблиць, statement timeout 10 с, ліміт рядків результату, заборона множинних statement-ів.

## 5. Архітектура та стек

```
[React SPA: чат, графіки, "How I answered"]
        │ SSE/streaming
[FastAPI]
  ├─ Router (Claude, structured output: {routes[], reasoning, clarify_question?})
  ├─ SQL Executor (readonly Postgres role) ──> demo_company schema
  ├─ RAG Retriever (pgvector, hybrid: vector + keyword)
  ├─ Web (Claude web search tool)
  └─ Synthesizer (зведення результатів маршрутів у відповідь + chart spec)
[Supabase]: demo-дані, embeddings, історія чатів, trace-логи
```

- **Frontend:** React + Vite + TS + Tailwind + Recharts (Netlify).
- **Backend:** Python/FastAPI (реюз інфраструктури DocFlow: той самий хостинг, спільні утиліти LLM-викликів, tracker).
- **Embeddings:** модель ембедингів обрати на етапі реалізації (порівняти 2 варіанти на власному retrieval-eval); зберігання — pgvector.
- **Trace:** кожен запит користувача → `trace(id, question, routes, steps jsonb, total_cost, total_latency)` — це і джерело для F7, і матеріал для evals.

## 6. AI-компоненти

| Компонент | Ключові вимоги | Промпт |
|---|---|---|
| Router | температура 0; structured output; few-shot по 3 приклади на маршрут, включно з mixed і clarify | `prompts/router.v1.md` |
| SQL Generator | у контексті — DDL схеми + описи колонок + 5 прикладів пар питання→SQL; тільки SELECT | `prompts/sql_gen.v1.md` |
| RAG Answerer | відповідь лише з наданих чанків; формат цитат; явна відмова за браку контексту | `prompts/rag_answer.v1.md` |
| Synthesizer | розділення internal/external фактів; рішення про chart spec | `prompts/synth.v1.md` |

## 7. Evals

- **Router eval:** 60 питань з розміткою правильних маршрутів (через Golden Dataset Builder). Метрика: exact route-set match + окремо частка небезпечних промахів (SQL-питання пішло у RAG тощо).
- **SQL eval:** 30 питань з еталонними результатами (не еталонним SQL — порівнюємо результат виконання). Метрики: виконуваність, правильність результату.
- **RAG eval:** 25 питань (20 з відповіддю в документах, 5 без). Метрики: правильність, groundedness, коректність відмов.
- `make eval` — всі три набори, зведена таблиця, порівняння з попереднім запуском.

## 8. Етапи реалізації

1. **Тиждень 1:** демо-дані (сідер), SQL-маршрут end-to-end з self-correction, базовий чат зі streaming.
2. **Тиждень 2:** RAG-маршрут (індексація, retrieval, цитати), Router, Synthesizer, графіки.
3. **Тиждень 3:** Web-маршрут, "How I answered", evals до цільових метрик, демо-режим, README, відео, деплой, публікації.

## 9. Definition of Done

- [ ] Три ключові сценарії (§2) працюють у живому демо
- [ ] Routing accuracy ≥ 92%, SQL-метрики і RAG groundedness — у README
- [ ] Блок "How I answered" показує повний trace кожної відповіді
- [ ] Заборонені операції (INSERT/UPDATE/DELETE, вихід за allowlist) блокуються — покрито тестами
- [ ] README (EN), відео ≤ 2 хв, 2 публікації

## 10. Ризики

| Ризик | Мітигація |
|---|---|
| Prompt injection через питання (спроба змусити виконати шкідливий SQL) | Захист на рівні прав БД, а не промпта; тести на injection-набір |
| Неоднозначні питання → хибний маршрут | Маршрут `clarify` як першокласний громадянин; окрема метрика в eval |
| Повільний mixed-маршрут | Паралельне виконання маршрутів (asyncio.gather) |
